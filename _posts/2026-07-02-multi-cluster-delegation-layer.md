## 배경: 애플리케이션이 k8s에 워크로드를 만든다

GPU 워크로드를 쿠버네티스(k8s) 클러스터에서 실행하는 서비스를 개발 및 운영하고 있습니다. 애플리케이션(백엔드)이 컨트롤플레인 역할을 맡아 사용자의 워크로드 생성·관리 요청을 받아 k8s API로 실행합니다. 애플리케이션이 곧 k8s API의 클라이언트입니다.

초기 오픈은 단일 클러스터였습니다. 애플리케이션이 클러스터 하나의 k8s API 서버와 통신하면 끝이었습니다. 둘이 동일 네트워크 안에 있어 도달성을 신경 쓸 일이 없었습니다.

## 문제: 단일 클러스터 전제가 깨졌다

오픈 이후 다음 버전을 준비하며 멀티클러스터 요구가 생겼습니다. 애플리케이션이 서로 다른 클러스터의 k8s API와 각각 통신해야 했습니다. "클러스터가 하나"라는 전제, 그 위에 얹혀 있던 "API 서버 주소도 하나"라는 전제가 함께 무너졌습니다.

## 대안: 설정을 늘릴까, 위임 계층을 둘까

가장 손쉬운 길은 클러스터별 k8s 접속 설정을 애플리케이션에 추가하는 것이었습니다. 클러스터가 둘, 셋이면 이것도 됩니다.

걸린 건 "클러스터는 계속 늘어난다"는 전제였습니다. 클러스터가 늘 때마다 kubeconfig·인증·엔드포인트 설정이 코드나 설정 파일에 쌓이면, 애플리케이션이 클러스터 개수와 접속 방식에 직접 결합됩니다. 결합도가 오릅니다.

헥사고날로 도메인을 포트 뒤에 두면 도메인은 k8s·전송 세부에서 분리된 채 유지됩니다. 그러나 애플리케이션이 모든 클러스터의 kubeconfig·자격증명을 직접 쥔다는 사실은 남습니다. 전 클러스터의 자격증명을 한 곳이 보유해 사고 범위가 커지고, 클러스터를 추가하려면 설정 변경과 재배포가 따라옵니다.

그래서 위임 서비스를 뒀습니다. k8s와 직접 말하는 책임을 위임 서비스로 옮기고 애플리케이션은 "어느 클러스터의 위임 서비스로 보낼지"만 알면 됩니다. 그 주소(클러스터별 위임 서비스 URL)는 DB 설정으로 관리합니다. **클러스터를 추가해도 애플리케이션 재배포 없이 DB 등록으로 끝납니다.** 각 클러스터의 자격증명도 애플리케이션이 아니라 해당 위임 서비스가 갖습니다. 어댑터는 클러스터별 k8s 접속 세부가 아니라 위임 서비스 계약 하나만 압니다.

### 또 하나의 이유: 실행 백엔드가 바뀔 수 있었다

당시 GPU 스케줄링은 k8s 대신 순수 docker + Slurm으로 전환하는 안도 검토 대상이었습니다. 실행 백엔드가 바뀔 수 있다면 애플리케이션이 k8s API에 직접 종속되면 안 됩니다. 그래서 워크로드를 생성할 때 "인프라에 보내는 요청"과 "그 요청을 실행 백엔드로 위임하는 일"을 나눠 추상화하기로 했습니다. k8s 관련 기능은 위임 서비스에 모으고 애플리케이션은 실행 기술에 독립적으로 남았습니다.

## 설계의 실체: 포트는 도메인에, 전송은 어댑터에

말로만 격리했다고 되는 게 아니라 코드에 이음새가 있어야 합니다. 도메인은 포트 인터페이스에만 의존합니다.

```java
// core: 도메인이 의존하는 포트 (전송·k8s 세부 없음)
public interface RemoteExecutionPort {
  void createWorkload(Workload workload);
  // ...
}
```

전송(HTTP)은 어댑터에만 있습니다. 어댑터가 `clusterId → delegatorUrl → client` 순서로 대상 위임 서비스를 해석합니다.

```java
// adapter: 포트 구현 = 전송 계층
public class RemoteExecutionAdapter implements RemoteExecutionPort {

  private DelegatorClient getClient(ClusterId clusterId) {
    var cluster = clusterQuery.getBy(clusterId);
    return delegatorClientFactory.getClient(URI.create(cluster.delegatorUrl()));
  }
}
```

동적 baseUrl은 팩토리가 클러스터별로 캐싱합니다.

```java
// infra: 위임 서비스 클라이언트 팩토리
public DelegatorClient getClient(URI baseUrl) {
  Objects.requireNonNull(baseUrl, "baseUrl must not be null");
  var normalizedUrl = normalizeUrl(baseUrl);
  return clientCache.get(normalizedUrl, url -> {
    var restClient = restClientBuilder.baseUrl(url).build();
    // ... HttpServiceProxyFactory 로 DelegatorClient 생성
  });
}
```

핵심은 여기입니다. 도메인은 `RemoteExecutionPort`에만 의존하고 전송·주소·k8s 세부는 전부 어댑터 밖에 있습니다. 그래서 전송을 gRPC 스트림이나 브로커로 바꿔도 **어댑터를 하나 더 구현할 뿐 도메인 코드는 그대로**입니다.

## 이 선택은 타당한가: 원격 통신 설계를 세 관심사로 점검

"위임 계층을 뒀다"까지는 좋습니다. 그런데 원격 노드와 통신하는 설계에는 흔히 세 가지가 뒤섞여서 "무엇이 무엇을 대체하나"에서 길을 잃습니다. 세 가지로 나눠 보면 정리됩니다. 표준 분류가 아니라 제가 정리한 구분입니다.

| 관심사 | 답하는 질문 | 대표 수단 |
|----|------------|----------|
| 발견(discovery) | 어떤 노드가 있고 주소는 무엇인가 | 자기등록, 서비스 디스커버리 |
| 전송·시맨틱 | 동기 명령인가 비동기 이벤트인가, 유실 가능인가 재생 가능인가 | REST·gRPC(라이브), 브로커(durable) |
| 도달성(연결 방향) | 누가 연결을 여는가, 방화벽·NAT를 넘는가 | 중앙 다이얼, 에이전트발 연결 |

브로커는 주로 전송·시맨틱을, 자기등록은 발견을, reverse tunnel은 도달성을 담당합니다. 다만 배타적이진 않습니다. 브로커는 양쪽이 아웃바운드로 붙는 구조라 도달성도 함께 해결합니다. 요점은 셋을 구분해 따로 판단하는 것입니다. 위임 서비스를 이 세 관심사로 따져 봅니다.

| 관심사 | 지금 구조 | 멀티리전으로 확장하면 |
|----|---------------|--------------------|
| 발견 | 위임 서비스 URL을 DB에 등록 (자기등록 없음) | 자기등록, 조인 토큰 |
| 전송·시맨틱 | 동기 RestClient (명령만, 역방향 이벤트 없음) | 명령은 유지, 상태는 이벤트 |
| 도달성 | 컨트롤플레인이 위임 서비스를 다이얼 (신뢰망 안에서만 성립) | 리전 격리 시 위임 서비스발 연결로 재설계 |

가장 약한 곳은 **도달성**입니다. 지금은 컨트롤플레인이 위임 서비스 주소로 직접 요청을 보냅니다(다이얼). 클러스터가 같은 신뢰망 안에 있으니 성립합니다. 그런데 리전이 갈려 위임 서비스가 방화벽·NAT 뒤로 가면 이 방향이 닿지 않습니다. 그때는 위임 서비스가 컨트롤플레인으로 지속 연결을 여는 역방향(reverse tunnel, gRPC bidi 등)으로 재설계해야 합니다. 예를 들어 쿠버네티스의 Konnectivity(apiserver-network-proxy), GitHub Actions 셀프호스티드 러너가 이 방식을 씁니다.

## 판단: 지금은 맞고 비싼 결정만 미뤘다

현 규모(신뢰망 안 소수 클러스터, 동기 명령 중심)에서는 위임 서비스 + 동기 REST + DB 등록이 맞습니다. 과한 것(멀티클러스터 서비스 메시, 브로커)을 미리 들이지 않았습니다.

비가역적(irreversible) 결정은 통신 방향(도달성)입니다. 이건 포트 뒤에 격리해 두고 멀티리전이 실제 요구가 되는 시점으로 미뤘습니다(last responsible moment). 그 시점엔 어댑터를 하나 더 구현하면 됩니다. 도메인은 그대로입니다.

"설정만 늘렸다면"과 비교하면 차이가 분명합니다. 설정 증설은 당장은 빠르지만 클러스터 증가와 실행 백엔드 변경을 코드나 설정에 계속 반영해야 합니다. 위임 서비스는 그 변화를 데이터(DB 등록)와 어댑터 교체 지점으로 모아 도메인이 흔들리지 않게 했습니다.

## 돌아보며

이 설계를 세 가지로 나눠 보기 전엔 통신을 한 덩어리로 봤습니다. 그래서 놓친 것들이 있었습니다.

- 도달성(연결 방향)을 전송·발견과 따로 세지 않았습니다. "브로커 하나로 다 풀린다"는 착각이 여기서 나옵니다.
- 방화벽 뒤 노드에는 에이전트가 역방향으로 연결을 여는 방식(reverse tunnel, gRPC bidi)이 있다는 걸 늦게 알았습니다.
- 전송을 도메인에서 분리하는 건 직관으론 했지만, 이음새·진화적 아키텍처라는 이름으로 설명하지 못했습니다.

세 가지로 나누고 나서야 지금 구조는 맞고 도달성 결정만 미뤘다는 판단을 말로 세울 수 있었습니다.
