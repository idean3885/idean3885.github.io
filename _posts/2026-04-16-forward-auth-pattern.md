> **TL;DR**<br>
> Forward Auth는 API Gateway가 요청을 백엔드로 전달하기 전에,
> 외부 인증/인가 서비스에 서브리퀘스트로 판정을 위임하는 패턴입니다.<br>
> 파일 스트리밍 등 데이터 전송 구간에 Auth 서비스가 끼지 않아
> 대역폭 병목이 발생하지 않습니다.
{: .prompt-tip }

## 패턴 정의

Forward Auth는 Gateway가 직접 인증/인가를 처리하지 않고,
별도 서비스에 "이 요청을 허용할까요?"라는 서브리퀘스트를 보내는 위임 패턴입니다.

핵심은 **데이터 경로와 인가 경로의 분리**입니다.

```text
[일반 프록시]
Client → GW → Auth → Backend     ← Auth가 데이터 경로에 있음

[Forward Auth]
Client → GW ─┬→ Auth (서브리퀘스트, 판정만)
              └→ Backend (원본 요청 전달)  ← Auth가 데이터 경로에 없음
```

## 동작 흐름

```text
1. Client → GW        : 원본 요청 도착
2. GW → Auth           : 서브리퀘스트 (헤더만 전달, 바디 미포함)
3. Auth → GW           : 200 OK (허용) 또는 401/403 (거부)
4. GW → Backend        : 허용 시 원본 요청 그대로 전달
```

- 2번에서 GW는 요청 바디를 Auth로 보내지 않습니다.
  헤더(Authorization, Cookie 등)만 전달합니다.
- Auth의 응답 헤더를 백엔드에 전파할 수 있습니다.
  (예: `X-User-Id`, `X-Permissions`)
- 거부 시 GW가 클라이언트에 직접 에러를 반환합니다.

## 스트리밍과 병목

Forward Auth에서 파일 업로드/다운로드 스트리밍 데이터는
Auth 서비스를 경유하지 않습니다.
GW가 Auth에 보내는 것은 헤더뿐이고,
인가 판정 후 원본 요청(바디 포함)은 백엔드로 직접 전달됩니다.

따라서 Auth 서비스의 처리량이 파일 전송 대역폭에 영향을 주지 않습니다.

## GW별 구현체

| Gateway | 구현체 | 타입 |
|---------|--------|------|
| APISIX | `forward-auth` | 플러그인 |
| Traefik | `forwardAuth` | 미들웨어 |
| NGINX | `auth_request` | 모듈 |
| Envoy | `ext_authz` | 필터 |
| Kong | `forward-auth` (커뮤니티) | 플러그인 |

이름은 다르지만 동작 원리는 동일합니다.
GW가 서브리퀘스트로 판정을 받고,
결과에 따라 원본 요청을 전달하거나 거부합니다.

## 적용 사례: 파일 서비스

실무에서는 이 패턴을 "쓸까 말까" 판단하기보다,
전제된 환경에서 어떻게 설계할까를 고민하는 경우가 더 많다고 봅니다.
이번 사례도 그랬습니다.
파일 서비스는 상위 플랫폼의 인가 체계를 재사용해야 하는 제약이 있었고,
그 제약상 API Gateway 레벨 forward-auth가 출발점으로 주어졌습니다.
선택한 것이 아니라 **주어진 환경에서 결정한 것**은 다음 네 가지입니다.

```text
Browser → LB → APISIX ─┬→ 플랫폼 Auth (인가 확인)
                        └→ 파일 서비스 BE (파일 전송)
```

### 인증/인가 주체 분리

인증은 BE가 맡고, 인가는 GW가 위임합니다.

- **인증(BE)**: 공통 JWKS(JSON Web Key Set)로 JWT(JSON Web Token)를 로컬 검증합니다.
  GW를 우회한 직접 접근에서도 미인증 요청을 BE가 차단합니다.
- **인가(GW)**: APISIX의 `forward-auth`로
  플랫폼 Auth에 경로 수준 퍼미션 판정을 위임합니다.

BE에 인증을 남긴 이유는 두 가지라고 봅니다.
상위 플랫폼의 다른 BE들도 JWT를 로컬 검증하므로 서비스 간 방식 일관성이 유지되고,
GW와 BE가 인증을 이중으로 확인해 단일 지점 우회가 성립하지 않습니다.

### 경로 규약의 제약

플랫폼 Auth는 경로 기반으로 인가를 판정합니다.
따라서 파일 서비스 API 경로가 상위 플랫폼의 경로 컨벤션에 맞아야 했습니다.
조직 및 프로젝트 식별자를 경로에 포함하는 형태가 강제됩니다.

이 제약은 "BE 내부에서 인가 로직을 유연하게 짜겠다"는 대안을 배제합니다.
경로가 판정의 입력이므로,
경로 설계 시점부터 상위 플랫폼의 인가 단위를 고려하게 됩니다.

### 라우트별 인가 정책 분리

FE 라우트와 BE API 라우트에 다른 정책을 적용합니다.

| 라우트 | 인가 | 이유 |
|--------|------|------|
| FE 도메인 (정적 리소스) | 없음 | FE가 로드되어야 접근 거부 화면을 표시할 수 있음 |
| BE API 도메인 | forward-auth | 데이터 접근은 BE API 경유이므로 여기서 보호 |

FE 요청에도 인가를 걸면 403이 발생했을 때 FE 자체가 로드되지 않아
"접근 거부" 화면조차 띄울 수 없습니다.
FE는 무인가로 로드되도록 두고,
BE API 호출 시점에 인가가 적용되도록 경계를 설정했습니다.
민감한 데이터는 전부 BE API를 경유하므로,
FE 무인가가 보안 공백이 되지는 않는다고 봅니다.

### 결과와 남은 과제

이 구성에서 얻은 것은 다음과 같습니다.

- **스트리밍 병목 회피**: 앞서 정리한 패턴 속성 덕에
  파일 전송량이 늘어도 플랫폼 Auth 부하와 무관합니다.
- **BE의 단일 책임 유지**: 파일 서비스 BE가 파일 I/O에 집중하며
  인가 로직 중복 구현이 없습니다.
- **이중 방어선**: GW의 인가 판정과 BE의 인증 검증이 독립적으로 동작해
  GW 우회 경로가 차단됩니다.

남은 과제는 아래와 같다고 봅니다.

- **FE 진입 가드의 위치**: 현재는 진입 시점 단일 API가 권한 확인 역할을 겸합니다.
  FE 페이지가 늘어나면 이 경로를 거치지 않는 진입이 생길 수 있어,
  미들웨어 수준으로 가드를 올리는 검토가 필요하다고 봅니다.
- **forward-auth 실패 정책**: 플랫폼 Auth 장애 시
  실패 시 차단(fail-close)으로 갈지 실패 시 허용(fail-open)으로 갈지는
  서비스 특성에 따라 달라집니다.
  이 사례에서는 차단을 기본으로 두었으나,
  유효한 세션이 일시적 Auth 오류로 끊기는 UX 비용도
  함께 고려해야 한다고 봅니다.
