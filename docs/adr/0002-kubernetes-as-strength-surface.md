# ADR-0002: Kubernetes를 별도 시리즈로 노출하여 강점 가시화

## Status

Accepted (2026-05-15)

## Context

저자는 K8s 환경에서 일하는 백엔드 개발자다. 인프라 담당자는 아니지만 K8s 운영 환경에 익숙하고 학습을 지속한다. 이 *온도*를 블로그 가시 표면에 어떻게 노출할지 결정이 필요했다.

- 기존 시리즈명 중 K8s 글에 후보로 검토된 자리: "개발 사전" 또는 "실무 노하우"
- 그러나 K8s 글을 "개발 사전"에 합치면 사이드바 펼침 메뉴에서 K8s가 별도 라벨로 노출되지 않아 강점 누적 효과가 사라진다
- K8s 글은 차근차근 매우 러프하게 쌓을 예정. 재직 기간 동안 누적 가능
- 향후 학습·운영 글이 섞여 들어올 가능성이 높음

## Decision

1. **기술 노하우** 상위 카테고리 정의를 *"운영 경험 기반 실전 지식"* → *"운영·학습 기반 실전 지식"* 으로 확장
2. 기술 노하우 하위에 **Kubernetes** 신규 시리즈 신설
3. **시리즈 인덱스(N편 구조) 미운영**. 카테고리 페이지(`/categories/kubernetes/`)가 자동 인덱스 역할

front matter 사용 예:

```yaml
categories: [기술 노하우, Kubernetes]
```

## Consequences

긍정:

- 사이드바 펼침 메뉴에 Kubernetes 라벨 독립 노출
- 카테고리 페이지가 시간 역순 자동 인덱스 역할
- 향후 다른 기술 학습 시리즈(예: Linux Networking, Database Internals)도 같은 둥지에 묶을 수 있음
- 학습·운영 글이 같이 들어와도 정의 어긋남 없음

부정:

- "기술 노하우" 정의가 약간 넓어짐
- 시리즈 인덱스가 없어 큐레이션은 카테고리 페이지 시간순에 의존

## Alternatives Considered

| 대안 | 기각 사유 |
|------|----------|
| 기존 "개발 사전" 시리즈에 K8s 글 합치기 | 사이드바에서 K8s 라벨 별도 노출 안 됨, 강점 누적 실패 |
| 신규 최상위 카테고리 "기술 학습" 신설 | K8s 1개 시리즈로 최상위 신설은 분류 비용 과함. 학습 시리즈 3개 이상 누적 시 재검토 가능 |
| 카테고리 정의 변경 없이 K8s 시리즈만 추가 | "운영 경험" 정의와 학습 글 성격이 어긋남 |
| 시리즈 인덱스(N편 구조) 운영 | "차근차근 쌓는다"는 의도와 충돌. 인덱스가 빨리 누더기가 되거나 방치 위험 |
